# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Horizon is Shopify's flagship Liquid theme (v3.4.0). It is a **server-rendered Shopify theme** — no React, no bundler, no Node.js build pipeline for the theme itself. HTML is generated by Shopify servers using Liquid; JavaScript enhances existing HTML using native web components.

## Commands

```sh
# Start local dev server (connects to a Shopify store)
shopify theme dev

# Lint and validate all theme files
shopify theme check

# Push theme to a Shopify store
shopify theme push

# Regenerate {% schema %} blocks in .liquid files from TypeScript source
npm run build:schemas
```

Theme Check runs automatically on every commit via GitHub Actions (`Shopify/theme-check-action`).

## Architecture

### Directory structure

| Path | Purpose |
|------|---------|
| `assets/*.js` | Web component implementations and utilities |
| `sections/` | Editable content sections (include `{% schema %}`) |
| `blocks/` | Reusable theme blocks |
| `snippets/` | Reusable Liquid partials |
| `templates/` | Page-type templates (product, collection, cart, etc.) |
| `layout/` | Root layout wrappers (`theme.liquid`, `password.liquid`) |
| `config/` | Theme settings schema and data |
| `locales/` | i18n translation files (30+ languages) |
| `schemas/` | TypeScript source for section/block schemas (compiled → `.liquid` files) |

### Component framework

All JavaScript uses a custom web component base class. Import from `@theme/component`:

```javascript
import { Component } from '@theme/component';

class MyComponent extends Component {
  connectedCallback() {
    super.connectedCallback();
  }
}

customElements.define('my-component', MyComponent);
```

`Component` (defined in `assets/component.js`) automatically:
- Wires up child elements with `ref="name"` attributes into `this.refs.name`
- Binds declarative event handlers: `on:click="/handleClick"` calls `this.handleClick(event)`
- Arrays of refs use `ref="items[]"` syntax

Components communicate via custom events with `bubbles: true`. Parents may call public methods on child components directly; children should emit events upward.

### Schema workflow — critical

**Never edit `{% schema %}` blocks directly in `.liquid` files.** Schemas are TypeScript sources in `schemas/blocks/` and `schemas/sections/`. Edit those, then run:

```sh
npm run build:schemas
```

Schema translation keys must exist in `locales/en.default.schema.json` under the `names` section before running the build.

## Coding standards

### Liquid

- Prefer inlining Liquid expressions directly in attributes/content over assigning intermediate variables — only assign when the same complex value is reused multiple times or cannot be inlined.
- Use `{% liquid %}` blocks for multi-line logic.
- Use `{% # comment %}` for inline comments.
- Never invent Liquid filters, tags, or objects that don't exist in the Shopify Liquid reference.

### Snippets

Every snippet must open with a `{% doc %}` / `{% enddoc %}` block documenting all parameters with `@param name {type}` and an `@example`.

### JavaScript

- **Zero external dependencies** — native browser APIs only.
- Always use `async/await` over `.then()` chaining.
- Use `for (const item of items)` over `.forEach()`.
- Use early returns over nested conditionals.
- Annotate with JSDoc `@typedef` and `@param` / `@returns` — no TypeScript files, but JSDoc drives type checking via `assets/jsconfig.json`.
- Use private class fields (`#field`) for internal state; only expose public methods for cross-component use.
- Use `const` by default; `let` only when reassignment is needed.

### CSS

- **BEM naming**: `.block`, `.block__element`, `.block--modifier`. Single element name per class; no chained elements like `__wrapper__button`.
- **Specificity**: target `0 1 0` (single class). Never use ID selectors; avoid `!important`.
- **CSS variables**: namespace scoped variables to their component (`--component-name-property`). Global variables live in `snippets/theme-styles-variables.liquid` scoped to `:root`.
- **Instance settings**: pass section/block settings as inline CSS variables on the element's `style` attribute, not via `{% style %}` with dynamic class names.
- **Logical properties**: use `padding-inline`, `margin-block`, `inset`, `text-align: start` etc. for RTL support.
- **Mobile-first**: use `min-width` media queries. Prefer container queries for component responsiveness.
- Never nest CSS beyond one level (media queries and direct modifier states are the exception).
- Use `transform`/`opacity` for animations; avoid animating layout properties.
- Always include `@media (prefers-reduced-motion: reduce)` fallbacks for animations.

### Accessibility

The theme enforces WCAG AA compliance. Key rules: all interactive elements need visible `:focus-visible` styles, never convey information through color alone, maintain 4.5:1 contrast for body text, and use semantic HTML landmarks correctly.
